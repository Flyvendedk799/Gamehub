Both load-bearing claims confirmed: `wantsVerdict` (lines 493-497) is gated on four declared capabilities, so a collectathon declaring none gets no `debug-snapshot` warning — Step 7's assert-path promotion is indeed inert for run2's class. The runtime `hasDebugContract===false` gate (line 651) is the real defense but only fires `if (hasPredicates)`. The churn math is exactly as cited (`rafChurn * 4`, particles/tweens default 0). Here is the final plan.

# 12-Step Sequenced Engine-Improvement Plan — "Lovable for Games"

**Core thesis of the sequence:** you cannot raise game quality faster than you can *measure* it, so the plan front-loads making the quality signals trustworthy and extending verdict coverage to the genres that ship blind, then turns warnings into bounded action, makes the always-on quality number a gradient, and only last reclaims the budget the skill apparatus wastes. The single goal underneath every step is the only one that matters: the built game must be good, and the engine must be able to *prove* it.

---

## 1. Stop the metrics from lying: widen the invariant + debug-wiring regexes — **Effort: S**
**Problem.** The static checks everything downstream trusts are false-positive machines. run3 (TD) shipped `debugWired=0` and `invariantWarnings=[score-or-state, debug-snapshot, …]` yet PASSED 4/4 predicates that *read* `score`/`wave` at runtime — proof the game *did* wire its snapshot and *did* score; the regexes just didn't recognize the style. `SCORE_PATTERNS` (assert-game-invariants.ts:200-204) require a literal-digit RHS, so `score = score + enemy.value`, `this.score += pts`, `kills = kills + 1` all miss. `SNAPSHOT_WIRING_PATTERNS` (:244-248) match only `debug.track(`, `debug.snapshot =`, whole-object `__game.state =` — missing `__game.state.score = x`, `__game.state['score'] =`, `Object.assign(__game.state, …)`, aliased `dbg.track(…)`. The same brittle set is duplicated in skill-usage-grep.ts:20-24 (`DEBUG_WIRING_PATTERNS`), so the build-report `debugWired` number is wrong too.
**Fix.** Add the missing forms to `SCORE_PATTERNS` (non-numeric RHS: `/\b(this\.|state\.)?(score|points?|coins?|kills?|money|lives)\s*[+\-]?=\s*[^=;]/i`) and `SNAPSHOT_WIRING_PATTERNS` (`/__game\s*\.\s*state\s*\.\s*\w+\s*=[^=]/`, `/__game\s*\.\s*state\s*\[/`, `/Object\.assign\s*\(\s*[^,]*__game\s*\.\s*state/`, `/\.\s*track\s*\(\s*\{/`). Mirror the snapshot edits into skill-usage-grep.ts:20-24. **Also reconcile the contradiction itself:** when the build report's `debugWired=0` but the runtime resolves `hasDebugContract===true` (run3 exactly), emit a single `wiring-style-unrecognized` telemetry note rather than silently disagreeing — a divergence here means a wiring form the regex still misses, which is a free lead for the next regex pass.
**Impact.** Foundational: every later gating/repair/cleanup decision (Steps 6, 7, 11) reads these. Today they would mis-gate good games and they train the agent to ignore *all* invariant warnings (it correctly ignored run3's false ones).

## 2. Tell 3D games to wire the verdict contract (one prompt section) — **Effort: S**
**Problem.** `three-engine-guide.v1.txt` — the always-on prompt for every `three` run — has *zero* mention of `debug.snapshot`/`debug.track`/the verdict contract (grep confirmed: none), while it *does* instruct games to build `yaw -= movementX * s`. Phaser runs lean on imported skills auto-wiring `getState()`, but run2 imported nothing (`engineImports=0`). So a 3D game is never told the one line that makes any predicate readable.
**Fix.** Add a short "Verification contract (REQUIRED)" section to three-engine-guide.v1.txt mirroring game-workflow.v1.txt, naming the canonical 3D fields the playbooks will read: `window.__game.debug.track({ playerPos: thePlayer.position, cameraYaw: () => yaw, score: () => score })` at startup.
**Impact.** High and unblocking: this single line is the precondition that makes Steps 3–4's new 3D predicates resolvable instead of reading `undefined`. Naming `playerPos`/`cameraYaw` here fixes the canonical field names the next steps must agree on.

## 3. Close the FPS/PUZZLE predicate gap → no more guaranteed `no_verdict` — **Effort: S–M**
**Problem.** FPS and PUZZLE are the *only* playable genres whose playbooks carry **zero** machine-checkable predicates (playtest-playbooks.ts:167-202 FPS, :204-232 PUZZLE — all steps are English `assert` strings). `selectGamePlaytestPlan('fps')` flattens to `predicates:[]` → `hasPredicates=false` (run-generation.ts:607) → `observeVerdict` returns null at the `fatalErrors.length===0 && !hasPredicates` line (:641) → `shipReason='no_verdict'`. This is exactly run2's signature (3D collectathon classified `fps`: NO_VERDICT, playbook 0/0). The engine literally cannot tell a working 3D game from a broken one.
**Fix.** Add keyboard-checkable predicates that don't depend on pointer-lock: FPS step-4 (`KeyW`) → `playerPos.z`/`playerPos.x` `op:'changed'` vs baseline + a grounded/no-fall check; add a keyboard-yaw step (`KeyQ`/`KeyE` or arrows) with `cameraYaw` `op:'changed'`. PUZZLE: `score` `op:'increased'` after a match. Mouse-look stays English-only (Step 9 backs it later).
**Impact.** High. Converts run2's class from "ships blind" to a real pass/fail, and routes FPS/PUZZLE through the *existing* `hasDebugContract===false` hard-repair at run-generation.ts:651 — so a broken 3D core now gets the bounded repair loop that already fixes phaser. Depends on Steps 1–2 (fields wired, debug-contract check not false).

## 4. Add the `collectathon` genre + its keystone "count rises on pickup" predicate — **Effort: M**
**Problem.** The genre run2 was *about* — a 3D collect-em-up — doesn't exist in `GameGenre` (game-spec.ts:19-38: platformer…fps/tps…tower_defense…idle/other; note the topdown token is `topdown_arcade`; no `collectathon`). It was force-fit to `fps`, the worst-covered genre. The single most natural deterministic check (`itemsCollected`/`score` increases on pickup) is never authored. `tycoon` (line 34) is a parallel gap — in the enum, but `selectGamePlaytestPlan('tycoon')===null`.
**Fix.** Add `collectathon` to the enum and a COLLECTATHON playbook (playtest-playbooks.ts PLAYBOOKS map): wait → move toward a pickup (`playerPos` changed) → keystone `score`/`itemsCollected` `op:'increased'`, modeled on TOWER_DEFENSE's place→kill→score keystone. Add a tycoon playbook or document it as requiring `declare_playtest_contract`.
**Impact.** High. Turns the actual flagship product use-case (3D collect-em-ups) into a first-class verified genre instead of an invisible `fps` fallthrough. Reuses Steps 2–3's field-wiring + predicate machinery, so it completes genre coverage right after the hole it patterns from.

## 5. Make the `wantsVerdict` gate fire for predicate-bearing genres + interactivity floor — **Effort: M**
**Problem.** Two coupled holes leave the flagship case unjudged. (a) The `debug-snapshot` warning is **capability-gated** (assert-game-invariants.ts:493-497): it fires only when `caps.hasFailState || hasEnemies || escalates || hasProgression`. A 3D collectathon that declares none of these (run2's exact class) gets **no** `debug-snapshot` warning at all — so any promotion of that warning (Step 7) is inert for the genre the plan most wants to fix. (b) Even with Steps 3–4, `no_verdict` stays reachable for any genre-less game whose agent skips `declare_playtest_contract`; the COVERAGE-GAP line logs but doesn't change the outcome. The runtime `hasDebugContract===false` gate (run-generation.ts:651) is the real backstop but only fires `if (hasPredicates)`.
**Fix.** (1) **Broaden `wantsVerdict`** so it also returns true whenever the resolved genre playbook has predicates (i.e. the game committed to measurable gameplay), not only on the four declared caps — this makes the snapshot discipline reach the collectathon. (2) Make `declare_playtest_contract` **mandatory** when `selectGamePlaytestPlan(genre).predicates` is empty (fix the now-wrong "SKIP it when a built-in playbook exists" guidance in game-workflow.v1.txt — FPS *had* a playbook that gated nothing). (3) Add a **universal interactivity floor** every booted game is scored against regardless of genre: "after N idle frames + one input burst, *some* tracked snapshot field changed." Its motion-area half is added in **5b inside Step 10** once `measureJuice` produces a real motion metric — Step 5 ships the interactivity half now and the motion half lands with Step 10 (sequenced below to avoid a measure-after-improve violation).
**Impact.** High. Eliminates the "ships behavior-unjudged" outcome and makes Step 7's promotion actually reach run2's genre. Caps the verdict-coverage tier (Steps 2–5).

## 6. Reject vacuous PASSes: a predicate that read nothing is not a pass — **Effort: M**
**Problem.** A PASSED verdict on an unobservable game is the worst signal — a false positive. `buildRepairVerdict` (repair-loop.ts:132-148) can reach PASSED when predicates resolve `'changed'`/`'increased'` against an *equally-empty* baseline (`undefined` vs `undefined`). `evaluatePredicate` returns "field X is missing" (playtest-score.ts:201-204) but the aggregate doesn't distinguish "passed on observed data" from "could not be evaluated." The runtime `hasDebugContract===false` gate (:651) catches most of this once Step 5 broadens its reach, but a *contract-authored* game with all-missing fields still slips through.
**Fix.** In `scorePlaytest`/`buildRepairVerdict`, add a third state — *unobservable* — distinct from pass/fail. When every predicate result is "field missing," treat the score as no-contract and route to the existing wire-debug repair instead of crediting passes. Add `predicatesActuallyObserved` vs `predicatesTotal`; require `observed > 0` for `shipReason='passed'`.
**Impact.** High. Closes the residual false-PASS loophole the runtime gate can't reach, so the verdict you now produce on every genre (Steps 3–5) is *true*. Sequenced **after** Step 5 (the broadened gate + debug-contract path are the primary defense; Step 6 catches only the residual contract-game case).

## 7. Promote `debug-snapshot` (and `controls`) to a bounded repair trigger — **Effort: M**
**Problem.** Behavioral invariant warnings are pure telemetry — they never gate `done` nor trigger repair (only `fail-state`/`restart`/`feedback` are in `FATAL_FLOOR_INVARIANTS`, :646-650). The self-defeating one is `debug-snapshot`: an unwired snapshot is the *exact* cause of `no_verdict`, so the engine warns about the cause of its own blindness and does nothing. `controls` (run3) is a second real, *deterministic* UX loss (keyboard read without `controls.define` via the KEYBOARD_INPUT_PATTERNS/CONTROLS_DEFINE check at :227-235 → unrebindable in the builder's Controls tab) that ships unaddressed. Meanwhile `score-or-state`/`escalation` recurred as **false positives** (run3 passed score-reading predicates), so promoting *them* would burn the bounded repair budget (default 2 / ceiling 3) on noise and could starve a real boot-fix.
**Fix.** Feed `assert_game_invariants` results into `observeVerdict`. When `debug-snapshot` is missing on a `wantsVerdict` game (now broadened by Step 5), push it as a fatalError — reusing the existing `hasDebugContract===false` mechanism (:651-655) and the staged-unused **one-shot** repair shape (:709-731): single capped round, only if budget+rounds remain, ordered *after* boot/playbook repairs. Add `controls` as a second low-false-positive one-shot in the same mechanism. Keep `score-or-state`/`escalation` advisory.
**Impact.** High. Converts run2/run3's "ship blind" into "wire the snapshot, then verify" — the one promotion that pays for itself because well-built games already wire it (≈0 extra rounds) and only broken ones spend the round; `controls` adds rebindable controls to the shipped product at near-zero false-block risk. Requires Step 1 (regex won't false-fire) and Steps 3–6 (real verdict + broadened gate for the snapshot to feed). This is where measurement finally becomes *action*.

## 8. Fix the escalation genre-map stub so genre-driven advisory checks are accurate — **Effort: S–M**
**Problem.** `mapSpecGenreToInvariantGenre` (done.ts:334-337) maps *only* `fighting→brawler`, returning `undefined` for all other genres. So at the `done` floor, genre-specific advisory checks never fire from the genre token — they rely entirely on the agent self-declaring `caps.escalates` (run2 only flagged escalation because the agent self-reported). The map is wrong regardless of whether escalation gates.
**Fix.** Extend the genre map: `shmup→shooter`, `runner→runner`, `tower_defense→tower-defense`, `collectathon→…`, etc., so `SHOULD_ESCALATE_GENRES` reflects the spec rather than the agent's self-report. Keep escalation **warn-only** (handcrafted level ramps make it a higher-false-positive signal — the plan does not gate on it). This is the *accuracy* fix for the advisory layer; the deterministic `controls` repair that used to anchor a standalone step has been folded into Step 7 where it belongs.
**Impact.** Medium. Makes the advisory layer genre-aware so warnings reflect the actual spec, without false-block risk. A small, same-file follow-on to Step 7; lower leverage (advisory, not verdict-blocking), so it sits just after the promotion that actually unblocks verification.

## 9. Synthesize relative pointer-lock input so mouse-look becomes testable — **Effort: L · first to defer if budget is tight**
**Problem.** The harness can't drive the input FPS games actually need. The synthetic-input union (playtest-planner.ts:128-133 and browser-worker main.ts:435-440) has only `key | mouseMove(absolute) | mouseDown | mouseUp | wait` — no relative pointer-lock delta. `projectPlaybookStep` maps FPS `movementX:50` to an absolute `mouseMove` at screen-centre; the worker dispatches `page.mouse.move(x,y)` (main.ts:944-950), never emitting `movementX`. A `yaw -= movementX * s` handler reads `movementX===0`, so a mouse-look yaw predicate could never pass — which is *why* the authors left FPS English-only.
**Fix.** Add a `mouseLook` (relative-delta) variant to both unions, dispatched via CDP `Input.dispatchMouseEvent` with `movementX/movementY` after a synthetic `canvas.requestPointerLock()` (Playwright's high-level API can't; CDP can). Then `projectPlaybookStep` faithfully projects FPS look steps and the mouse-yaw predicates become testable.
**Impact.** Medium — but the highest-effort step serving the narrowest axis (mouse-look yaw on FPS/TPS only). Step 3 already converts FPS to keyboard-checkable pass/fail and Step 5's floor catches interactivity, so the marginal verdict gain is small relative to L cost. First *structural* change, deliberately deferred behind the cheap keyboard-predicate path; **explicitly the first candidate to cut if the slate must shrink.**

## 10. Turn `juiceScore` from a floor into a real quality gradient (+ Step 5's motion half) — **Effort: M**
**Problem.** `juiceScore` saturates at 384–385 across a platformer, a 3D collectathon, and a TD — identical regardless of richness. The math: `rafChurn = min(rafCount, frameWindow*4)` → `churn = rafChurn*4 + particles + tweens*2` → a bare RAF loop alone produces ~384 (browser-worker main.ts:674-678, confirmed); the async-decode fallback credits `pixelDelta += 1` (:663-668) → 385. `particles`/`tweens` are *always* 0 — nothing in any prompt/template/skill tells a game to expose `debug.particleCount`/`activeTweens` (confirmed at :676-677). For a `no_verdict` 3D run this saturated number *is* the quality bar.
**Fix.** Rebuild `measureJuice` into a graduated multi-component score: (a) sample pixel-delta *magnitude* + spatial spread (changed quadrants) so a full-screen explosion outranks a cursor blink; (b) fix the decode path with `img.decode()` so `pixelDelta` is real motion area, not a +1 fallback; (c) add one line to engine starter templates + a prompt directive so every game exposes `debug.particleCount`/`activeTweens`; (d) raise/remove the `rafChurn` sub-cap and re-weight to a ~0..several-thousand range; (e) surface `juiceFloorOnly=true` when churn dominates and pixelDelta is low; persist the component breakdown in `run_quality_metrics`. **Then land Step 5b:** wire the now-real motion-area metric into the universal floor as the "motion-area > floor" half (proves the game renders motion), completing the floor that Step 5 shipped interactivity-only.
**Impact.** High. Makes the one always-on quality number actually *rank* games — the anti-slop differentiator the workflow spends a section on, currently unmeasured — and completes Step 5's floor. Sequenced after the behavioral-verdict tier (a gradient on the cosmetic axis matters less than knowing the game *works*) but as the producer of the motion metric Step 5 needs, resolving the measure-after-improve ordering: the floor's motion half is *defined here*, not assumed earlier.

## 11. Stop spending budget chasing `usesSkillFns`; reclaim it for quality — **Effort: M**
**Problem.** `usesSkillFns=0` in all three runs — the curated library is never *called*. run1 imported `level-orchestrator` and never wired it (`skillImportedNotCalled` → `skill-staged-unused` shipped as dead bytes the user paid tokens for); runs 2/3 imported nothing. The prior matched study found skill adoption *doesn't* raise PLAY quality. So the whole apparatus (recommend-skills push at choose-engine.ts:110-119, `import_skill` nagging in game-workflow.v1.txt, the staged-unused repair round, the dead-skill sweep) spends tokens, prompt real-estate, and one of only 2–3 repair rounds chasing a metric that doesn't move quality — and the worst outcome (import-then-not-use) is what run1 produced.
**Fix.** (1) Retire the import_skill *push*: drop the recommendation injection (choose-engine.ts:110-122) and the nagging paragraphs in game-workflow.v1.txt. (2) Keep import_skill as a *pull* tool, but stop counting `recommendedButUnused`/`importWithoutUse` as a quality signal (run-report-analysis.ts) — track `usesSkillFns` as telemetry only. (3) Redirect the freed repair round + prompt budget to completability + the real verdict (Steps 3–7). (4) Cheap cleanup: extend the dead-skill sweep (run-generation.ts:748-763) to also strip a module that's imported-but-never-called whose sole reference is the unmodified auto-wire stub, so no staged-unused dead code ships even when the repair round is skipped. (5) **Fold in the only durable part of the deprecated inline-rewrite idea:** fix the ordering so any remaining import nudge fires *after* `src/main.js` is scaffolded (today it nudges at choose_engine time, before the entry exists, so the auto-wire can't attach and the agent hand-rolls — run1's staged-unused).
**Impact.** Medium (neutral-to-positive on quality, frees budget). Converts a token sink into headroom for the verdict/quality work and removes the import-then-abandon waste entirely. Comes after the verdict + gradient tiers because its value is *redirecting* the freed round toward them (Steps 7, 10) — there must be a better place to spend it first. Depends on Step 1 (so `skillImportedNotCalled`/`debugWired` are accurate before you stop acting on them).

## 12. Make the library earn its keep through behavior, not import counts — **Effort: M**
**Problem.** After Step 11 retires the push, one structural root of zero adoption remains: the library's value is still framed as an *import count* rather than landing through the *verdict*. The skills that genuinely help (wave-spawner→escalation, camera-controller→camera-relative, debug helpers→debug-snapshot) overlap *exactly* with the unmet invariants — so if behavior is what's gated, the library becomes optional-but-cheap rather than dead-weight-when-imported. (The inline-into-main.js rewrite considered here was cut: the matched study says adoption doesn't move quality, so an M–L rewrite of `import_skill` to append function bodies — with its name-collision and boot-fragility risk, the very file-count hazard game-workflow.v1.txt:13 warns about — is gold-plating a deprecated subsystem.)
**Fix.** Do *not* rewrite import_skill. Instead, make the curated skills satisfy the **behavior gates** Steps 7–8 introduced (escalation present, camera-relative present, debug wired), satisfied however the agent likes — hand-rolled or imported. Document each skill by the invariant it discharges, so the agent reaches for it as a shortcut to passing a gate, not to hit a count. Leave import_skill a pure pull-tool with telemetry only (per Step 11). This is the terminal "make the library optional and stop measuring it" move.
**Impact.** Medium-to-low on direct quality, but it removes the last structural source of waste and reorients the subsystem around outcomes. Last because its payoff *depends* on the behavior gates from Steps 7–8 being live and on Step 11 having removed the push — lowest urgency relative to the verdict/quality work, correctly the tail.

**Sequencing rationale.** The order is a strict dependency-plus-leverage gradient anchored to one rule: fix the ruler before you measure, and prove the game works before you grade how it feels. Steps 1–2 make the signals truthful and tell 3D games how to be observed (S, prompt/regex, zero structural risk) — and they are hard preconditions: Step 1's regex feeds every later gate, and Step 2 names the `playerPos`/`cameraYaw` fields that Steps 3–5 must read. Steps 3–5 extend verdict *coverage* outward from the genres that ship blind, each reusing the prior step's field-wiring and predicate machinery, with Step 5 broadening the `wantsVerdict` gate so the promotion in Step 7 isn't inert for the flagship collectathon. Step 6 then makes those now-universal verdicts *honest* (no vacuous PASS), correctly placed after Step 5's broadened debug-contract path so it only mops up the residual case that path can't reach. Step 7 is the hinge — measurement becomes bounded action — and it must follow 1 (no false-fire) and 3–6 (a real verdict to feed); Step 8 is its low-risk same-file accuracy follow-on. Steps 9–10 are the larger verification-infra and quality-*gradient* upgrades, deliberately deferred behind the cheap coverage wins; Step 10 is placed to *produce* the motion metric that Step 5's floor consumes, resolving the only genuine ordering hazard by defining the metric where it is built rather than assuming it earlier. Steps 11–12 reclaim and re-point the skill budget last, because their entire payoff is feeding the earlier tiers — you free the repair round only once there is a better place (Steps 7, 10) to spend it, and you reorient the library around the behavior gates only once those gates exist. Throughout, the held goal is the one that matters: the built game must be good, and the engine must be able to prove it.
